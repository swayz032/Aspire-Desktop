## What Changed

### Summary
A new `POST /api/service-hub/property-data` Express endpoint aggregates property data from four sources (Google Address Validation, Adam/ATTOM+Apify, Google Geocoding, Google Solar) and caches results per-tenant in a new `property_snapshots` Supabase table.

**10 new TypeScript files** in `server/serviceHub/property/`.
**3 new Python files** in `backend/orchestrator/src/aspire_orchestrator/` (providers + normalizers + trades playbook extension).
**1 new Supabase migration** (`20260510170000_property_snapshots.sql`).
**2 files modified**: `server/index.ts` (mounts route), `server/receiptService.ts` (adds optional `status` + `correlationId` params).
**2 legacy files deleted**: `apifyZillowClient.ts`, `googleStaticMapsClient.ts`.

### Risk Classification: GREEN

- No financial operations, no contract signing, no irreversible actions.
- The endpoint is read-only from the user's perspective (property data lookup + cache write). Cache is ephemeral (24h TTL).
- Receipts written for every outcome — no state change without a trace.
- Failure mode degrades gracefully to empty `evidenceGaps[]` response; never returns invented data.

### Blast Radius
If this endpoint fails completely:
- The Visuals tab shows empty property data (photos, facts, signals all blank). User experience degraded but no data loss.
- No other endpoint is affected. The route is mounted independently (`require('./serviceHub/property/propertyRoutes').default`) at `server/index.ts` line 718-719.
- Receipts service and all other routes are unaffected.
- The `receiptService.ts` change is additive-only (optional params with defaults); no existing callers break.

## Operational Runbook Stub

See `docs/runbooks/service-hub-property-data.md` for full runbook.

### Quick reference — how to detect issues

**Slow property fetch (>5s user-perceived)**
1. Check Adam timeout: `ADAM_TIMEOUT_MS` is 25,000ms. If Adam is consistently hitting this, Apify cold-start is the likely cause.
2. Check Apify: `curl -H "Authorization: Bearer $ASPIRE_APIFY_API_KEY" https://api.apify.com/v2/users/me` — verify token valid + no quota exhaustion.
3. Check receipts: `SELECT * FROM receipts WHERE action_type='compute_snapshot' AND created_at > now() - interval '1 hour' ORDER BY created_at DESC LIMIT 20;`

**Cache poisoned (stale/wrong data for an address)**
```sql
DELETE FROM public.property_snapshots
WHERE suite_id = '<suite_uuid>'
  AND address = '<address>';
```

**Verify receipt chain for a request**
```sql
SELECT receipt_id, status, correlation_id, inputs, outputs, created_at
FROM receipts
WHERE correlation_id = '<correlation_id_from_response_or_log>'
ORDER BY created_at;
```
